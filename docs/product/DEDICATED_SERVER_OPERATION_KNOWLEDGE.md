# Dedicated Server 운영 지식

이 문서는 Core Keeper Dedicated Server 자동화 개발과 운영 기능 설계에 참고할 확인된 사실과 검증 필요 항목을 정리한다.

## 확인된 사실

### 플레이어 0명일 때 서버는 Sleep/대기 모드로 전환

Core Keeper Dedicated Server에 아무도 접속하지 않으면 서버는 자동으로 sleep 또는 대기 모드로 전환되는 것으로 확인되었다.

운영상 의미:

- 친구들이 접속하지 않는 동안 서버 프로세스는 떠 있어도 월드 시뮬레이션은 계속 진행되지 않는 방향으로 봐야 한다.
- “아무도 접속하지 않아도 농작물, 자동 채굴, 컨베이어, 용광로, 몬스터 AI가 계속 진행된다”는 전제로 자동화 기능을 설계하면 안 된다.
- 24시간 Dedicated Server의 핵심 가치는 “언제든 접속 가능”이지, “무인 상태에서 월드 시간이 계속 흐름”이 아니다.

자동화 설계 영향:

- 플레이어 0명 상태에서 월드 진행 여부를 운영 기능으로 억지 제어하지 않는다.
- 자동 백업은 플레이어가 없는 시간대에 실행해도 월드 진행 중 데이터 변경이 적을 가능성이 있지만, 실제 저장 타이밍은 Windows 실기 검증으로 확인한다.
- Watchdog은 “서버 sleep”을 장애로 오인하면 안 된다. 프로세스가 정상 실행 중이면 restart 대상이 아니다.
- 상태 조회 기능을 확장할 때 `Running`, `Sleeping/Idle`, `Players Online` 같은 구분을 둘 수 있는지 검토한다.

## 1차/2차 분리 원칙

- 1차 범위는 T1-T9 구현과 T10 Windows 실기 검증이다.
- T10은 설치/실행 성공 여부 검증과 함께 2차 운영 확장에 필요한 실제 로그와 파일 구조를 수집한다.
- 자동 백업 정책, Watchdog, Discord Webhook, 상태 조회 확장, 실제 자동 재시작은 T10 결과를 근거로 설계한다.
- T10에서 확인되지 않은 콘솔 명령, 로그 형식, 저장 타이밍을 추정해서 자동화하지 않는다.

T10에서 수집할 핵심 증거:

- 서버 실행 파일명 또는 배치 파일명
- 서버 시작 콘솔 출력과 Game ID 출력 위치
- Dedicated Server 데이터 경로와 `ServerConfig.json` 실제 구조
- 서버 로그 파일 위치와 플레이어 접속/퇴장 로그
- 플레이어 0명 sleep/idle 진입 및 재접속 복귀 로그
- 안전 종료 방식과 종료 시 저장 완료 시점
- 서버 실행 중 백업 안전성
- Task Scheduler 자동 실행/예약 작업 실제 동작

## 조사 요청에서 도출된 검증 항목

다음 항목은 운영 기능 확장 전 Windows Dedicated Server에서 실제 확인해야 한다.

### 월드 동작

- 플레이어 0명일 때 월드 시간이 정지하는지
- 농작물 성장, 드릴, 컨베이어, 용광로, 자동화 설비가 멈추는지
- 몬스터 스폰과 AI가 멈추는지
- 일반 플레이어 호스트 방식과 Dedicated Server 방식의 차이

현재 판단:

- 서버가 sleep/대기 모드로 전환되므로 무인 상태 월드 진행을 기대하지 않는다.
- 세부 설비별 동작은 T10 이후 실험 또는 신뢰 가능한 자료로 보강한다.

### 저장 시스템

- 자동 저장 존재 여부
- 저장 주기
- 수동 저장 기능
- 정상 종료 시 저장 방식
- 비정상 종료 시 저장 손실 가능성
- 업데이트 전 권장 백업/종료 순서

현재 판단:

- 현재 스크립트는 안전 종료 방식이 미검증이므로 강제 종료를 하지 않는다.
- 업데이트 전 백업은 이미 지원한다.
- graceful shutdown 방식 확인 전에는 자동 업데이트/자동 재시작을 공격적으로 구현하지 않는다.

### 캐릭터와 월드 데이터

- 캐릭터 저장 위치
- Steam Cloud 사용 여부
- 인벤토리/장비/스킬/사망 정보 저장 위치
- 월드 파일과 캐릭터 파일의 분리 여부

현재 판단:

- 현재 자동화는 서버 월드 파일만 다룬다.
- 플레이어 캐릭터 데이터는 각 플레이어 클라이언트/Steam 계정 영역에 속하는 것으로 보고 서버 자동화 범위에서 제외한다.
- 기존 월드 import는 단일 `.world.gzip` 파일 기준으로 유지한다.

## 운영 기능 확장 후보

두 product 요구사항 문서에서 나온 기능을 아래 우선순위로 관리한다.

### Priority 1 후보

- 서버 시작 정보 출력
- 상태 조회 확장
- 자동 백업 정책
- Watchdog 자동 재시작
- Discord Webhook 알림

### Priority 2 후보

- 백업 복원
- 최신 백업 확인
- 플레이어 접속/퇴장 알림
- 월드 상태 표시

### Priority 3 후보

- 자동 점검
- 자동 업데이트
- 서버 상태 페이지

## 기능별 설계 판단

### 서버 시작 정보 출력

요구 출력:

```text
World Name
World ID
Current Game ID
Save Path
Server Start Time
Max Players
```

현재 상태:

- `status-server.ps1`가 설치 경로, 데이터 경로, 실행 프로세스, Game ID 로그 힌트를 출력한다.

추가 설계:

- `CoreKeeper.Server.psm1`에 시작 정보 추출 함수를 추가하는 후보를 둔다.
- Game ID가 로그/파일 어디에 남는지 T10에서 확인한 뒤 구현한다.

### Watchdog 자동 재시작

요구:

- 기본 비활성화
- 비정상 종료 시 자동 재시작
- 정상 종료 시 재시작하지 않음
- 재시작 횟수와 시간 로그 기록

설계 주의:

- 플레이어 0명 sleep/대기 모드를 장애로 오인하면 안 된다.
- 정상 종료와 비정상 종료를 구분할 방법이 필요하다.
- 안전 종료 방식 확인 전에는 구현하지 않는다.

### 자동 백업

요구:

- 지정 시간 자동 백업
- 업데이트 전 자동 백업
- 수동 백업
- 최근 N개 보관
- 백업 복원

현재 상태:

- 수동 백업과 import 전 백업은 구현되어 있다.
- 업데이트 전 백업은 명령으로 실행 가능하다.

추가 설계:

- Task Scheduler 기반 일일 백업 작업을 후보로 둔다.
- 보관 개수 기본값은 예: 최근 30개 후보로 검토한다.
- 복원은 destructive 작업이므로 별도 확인 단계를 둔다.

### Discord Webhook 알림

요구 이벤트:

- 서버 시작
- 서버 종료
- 서버 재시작
- 서버 충돌 감지
- 백업 완료
- 업데이트 시작/완료

설계 주의:

- Webhook URL은 비밀값이므로 `settings.local.json`에만 저장한다.
- 절대 커밋하지 않는다.
- 알림 실패가 서버 실행을 막으면 안 된다.

### 상태 조회 기능

요구 출력:

```text
Server Status
Uptime
Game ID
Restart Count
Last Backup
```

현재 상태:

- `status-server.ps1`가 기본 상태 조회를 제공한다.

추가 설계:

- Last Backup과 Uptime은 로컬 파일/프로세스 정보로 구현 가능성이 높다.
- Game ID는 T10에서 실제 출력 위치 확인 후 안정화한다.
- Players Online과 Sleep/Idle 상태는 로그 또는 서버 API가 있어야 가능하다.
